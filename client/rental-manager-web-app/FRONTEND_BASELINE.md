# Frontend baseline

This document is the copy/reference checklist for every new Rental Manager feature.

## Required resource structure

```text
src/api/routes/<resource>/
├── types.ts       # API request and response contracts
├── requests.ts    # Axios calls only
├── queries.ts     # query keys and queryOptions
├── hooks.ts       # useQuery/useMutation and cache invalidation
└── index.ts       # public exports

src/features/<resource>/
├── components/    # forms and business-specific UI
└── pages/         # page composition only
```

## Required route structure

Routes use TanStack file-based routing. Do not add new route definitions to one central route file.

```text
src/routes/
├── __root.tsx
├── index.tsx
├── _authenticated.tsx
└── _authenticated/
    ├── access-denied.tsx
    ├── _standard.tsx
    └── _standard/
        └── (tenant)/
            └── <resource>/
                ├── index.tsx
                └── $resourceId.tsx
```

Naming conventions:

- `_authenticated` and `_standard` are pathless layouts.
- `(tenant)` is a route group and does not appear in the URL.
- `index.tsx` represents the resource list page.
- `$resourceId.tsx` represents a dynamic detail route.
- `routeTree.gen.ts` is generated by `npm run routes:generate`.

## Responsibilities

| Layer | Allowed responsibility |
| --- | --- |
| `components/ui` | shadcn primitives |
| `components/form` | React Hook Form wiring, labels and errors |
| `components/common` | generic application patterns |
| `api/client.ts` | HTTP transport and headers |
| `api/routes/*/requests.ts` | endpoint calls |
| `api/routes/*/queries.ts` | query keys and cache definitions |
| `api/routes/*/hooks.ts` | mutations, invalidation and cache updates |
| `api/mocks` | local Faker data and endpoint interception |
| `features/*` | rental business composition |
| `routes/*` | URL schema, loader, auth and selected page component |

## List page flow

```text
URL search params
      ↓
TanStack Router validateSearch
      ↓
route loader ensureQueryData
      ↓
resource queryOptions
      ↓
request function
      ↓
Axios client
      ↓
Mock adapter or .NET API
```

Use the generated route API inside a page:

```ts
const routeApi = getRouteApi(
  "/_authenticated/_standard/(tenant)/contracts/",
);

const search = routeApi.useSearch();
const navigate = routeApi.useNavigate();
```

Do not use an untyped global search state and do not duplicate URL parsing inside pages.

## Mock API rule

Development enables mocks by default.

```env
VITE_ENABLE_MOCK_API=true
VITE_MOCK_API_DELAY=250
```

The mock layer must:

1. Intercept the shared Axios instance.
2. Return the same contracts as the real backend.
3. Support filtering and server pagination for list endpoints.
4. Return `.NET ProblemDetails`-compatible validation errors.
5. Use a fixed Faker seed so baseline data is repeatable.
6. Stay outside feature pages and Query hooks.

Set `VITE_ENABLE_MOCK_API=false` when testing the real .NET API.

## Mutation flow

```text
Feature form
   ↓ mutateAsync(values)
resource mutation hook
   ↓
request function
   ↓
update detail cache + invalidate list cache
```

## New feature checklist

For a new resource such as `contracts`:

1. Copy `api/routes/rooms` to `api/routes/contracts`.
2. Replace contracts in `types.ts`.
3. Replace URLs in `requests.ts`.
4. Keep the query key factory pattern in `queries.ts`.
5. Configure invalidation in `hooks.ts`.
6. Add a Faker factory and mock handlers when backend APIs are unavailable.
7. Create `ContractForm` using fields from `components/form`.
8. Create the page using `PageHeader`, `SearchInput`, `DataTable` and page states.
9. Add an `index.tsx` route with search schema and route loader.
10. Add `$contractId.tsx` when a detail page is required.
11. Add required permission keys to `PermissionGuard`.
12. Run route generation, type-check and build.

## Multi-tenant rule

Never reuse cached data after an organization switch.

```ts
await setOrganizationId(nextOrganizationId);
```

The helper clears Query cache and invalidates the Router before the new organization view is rendered.

## Permission rule

`PermissionGuard` only controls visibility and UX. Every API endpoint must independently enforce its permission.

## Validation rule

Feature schemas own validation. Components render errors but do not duplicate business validation rules.

Server validation follows `.NET ProblemDetails` and is mapped centrally by `AppForm`.
